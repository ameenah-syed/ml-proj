# Eating Disorder Classifier

A research/educational ML pipeline that classifies simulated patients into
one of eight categories — `none`, `anorexia_nervosa`, `atypical_anorexia`,
`bulimia_nervosa`, `atypical_bulimia_nervosa`, `binge_eating_disorder`,
`arfid`, or `osfed_other` — from behavioral, weight-trajectory, vital-sign,
and lab features, all generated by a synthetic patient simulator instead of
real clinical data.

## Acknowledgment

Nearly every clinical threshold and piece of diagnostic reasoning in this
project's data generator is drawn from one book: Jennifer L. Gaudiani, MD,
FACP, CEDS-C, *Sick Enough: A Guide to the Medical Complications of Eating
Disorders* (Routledge, 2019). Dr. Gaudiani took a body of medical knowledge
that's often scattered across journal articles and clinical training and
turned it into something a non-specialist could actually read, follow, and
build from — this project exists because that information was made
digestible in the first place. Full citation and chapter-by-chapter credit
are in [Clinical grounding](#clinical-grounding) and [References](#references).

## Why this project exists

The project started narrower — a classifier for atypical anorexia nervosa
(AAN) specifically — and grew into a multi-disorder classifier from there.
AAN is still the flagship case study running through the design, because it
crystallizes the core argument of the book this project is built around
(Jennifer L. Gaudiani, MD, *Sick Enough: A Guide to the Medical
Complications of Eating Disorders*, Routledge, 2019): a patient who
restricts food, fears weight gain, and has a distorted body image is
**equally sick** whether or not their weight happens to fall under a BMI
cutoff — but the ones who aren't underweight get dismissed as "not sick
enough," by themselves, their families, and often their doctors, because
medicine over-indexes on body weight as a proxy for illness severity.

That argument only holds up if a classifier trained on this data can
actually reproduce it — i.e., if it learns to flag AAN using behavioral and
physiological signals rather than defaulting to "not underweight, so
probably fine." The whole feature-engineering and data-generation design in
this repo is built to make that a real test rather than a foregone
conclusion: AN and AAN patients are given **identical** behavioral-severity
distributions in the generator (see below), so the model has no shortcut —
it has to actually use weight-trajectory and physiologic features to tell
them apart, the same way a clinician has to.

The project later expanded to cover bulimia nervosa, binge eating disorder,
ARFID, and OSFED as well, since the book covers each with the same
"medical seriousness independent of appearance" argument, just with
different diagnostic mechanics (see [Clinical grounding](#clinical-grounding)).

## Important disclaimer

This project trains on **synthetic data only**. There is no real patient
data anywhere in this repo, and there shouldn't be — eating-disorder
records are sensitive protected health information, and no public dataset
of real patients exists for this project to have used even if that were
desirable. The model is a research and educational artifact for exploring
how weight-trajectory and physiologic features relate to eating-disorder
presentations; **it is not a diagnostic tool and must never be used to make
or support a clinical diagnosis.** Diagnosing any eating disorder requires
a full clinical assessment by a qualified provider. If you or someone you
know may have an eating disorder, the National Eating Disorders Association
helpline (nationaleatingdisorders.org) is a real, human resource — this
repository is not.

## How it works

The pipeline has four stages, each its own module, run in sequence:

**1. Synthetic data generation** (`src/data_gen.py` + `src/disorders/`)
Real patient records don't exist for this project to use, so every "patient"
is simulated. Generation works top-down:
  - Shared demographics (age, sex, height) are sampled once per patient.
  - Each patient is assigned one of the 8 labels, drawn from a configurable
    distribution (see [Usage](#usage) for the two distributions used —
    balanced-for-training vs. realistic-prevalence-for-evaluation).
  - Each label's patients are handed off to that disorder's module under
    `src/disorders/` (e.g. `anorexia.py`, `bulimia.py`), which samples
    behavioral scores, weight trajectory, binge/purge behavior, vitals,
    labs, and exam findings using distributions calibrated to match the
    clinical literature (see [Clinical grounding](#clinical-grounding)).
    Columns a given disorder doesn't use (e.g. `binge_frequency_per_week`
    for a pure restrictor) get a clinically defensible baseline value from
    `src/disorders/common.py`, not a missing value — every patient row has
    every column populated.
  - A per-patient "resilience" factor dampens (but never fully masks) how
    much illness severity shows up in vitals and labs, because the book is
    explicit that lab values can be near-normal in genuinely sick patients
    — a classifier that only worked when labs were dramatically abnormal
    would be learning the wrong lesson.

**2. Feature engineering** (`src/features.py`)
  Raw simulated chart values (e.g. `current_weight_kg`, `resting_hr`,
  `potassium_meq_l`) are turned into the derived clinical metrics an actual
  clinician would compute — percent median BMI, weight suppression
  percentage, the "walk across the room" heart-rate-rise test, true
  orthostatic vital sign criteria, hypokalemia/hypophosphatemia/QTc-
  prolongation flags, and more (see [Clinical grounding](#clinical-grounding)
  for the specific thresholds and their sourcing). These derived features,
  not just the raw numbers, are what gets fed to the models.

**3. Modeling** (`src/model.py`)
  Two models are trained: logistic regression and a random forest, both
  with `class_weight="balanced"`. Both were chosen for interpretability —
  logistic regression coefficients and random forest feature importances
  both stay human-readable, which matters for a project about medical
  features where "the model said so" isn't an acceptable justification on
  its own. Both share one `sklearn.compose.ColumnTransformer` preprocessing
  step (standard-scaled numeric features, passthrough booleans, one-hot
  categoricals).

**4. Evaluation** (`src/evaluate.py`)
  Reports per-class precision/recall/F1, macro-average ROC-AUC, confusion
  matrices, and feature importance plots. Supports two evaluation modes —
  balanced (matching the training distribution) and realistic-prevalence
  (matching true population rates) — run against the *same* trained models,
  so you can see both how well the model discriminates between classes and
  how it would actually perform if deployed against a real, heavily
  imbalanced population. See [Usage](#usage) for how to run both.

## Clinical grounding

The synthetic data generator (`src/disorders/*.py`) and feature engineering
(`src/features.py`) are built around specific concepts and numeric
thresholds from *Sick Enough* — cited throughout the code by chapter/page.
Key ideas used, by disorder:

**Anorexia nervosa (AN) and atypical anorexia (AAN)** — `src/disorders/anorexia.py`
- **Weight suppression** = (highest recent weight − current weight) /
  highest recent weight; even 5% is clinically significant in the presence
  of ED symptoms (Ch. 1).
- AN and AAN patients have **equally severe behavioral/psychological
  symptoms** — the DSM-5 distinction is weight status alone, not illness
  severity (Ch. 1). Both draw behavioral scores from an identical
  distribution so the model can't cheat by learning "severity implies
  underweight."
- AN's **restricting vs. binge/purge subtypes** (Ch. 1, p.818-829) are
  modeled as `purging_behaviors`/`binge_behaviors` features, not separate
  labels — purge-subtype AN still requires the underweight criterion
  (p.4561-4563).
- The **"walk across the room" test** (resting vs. ambulatory pulse after
  walking down the hall): a rise of ~75%+ with stable blood pressure is the
  "starving heart" pattern, distinct from true orthostasis (Ch. 2).
- **True orthostatic vitals** (lying vs. standing 3 min): SBP drop ≥20, or
  DBP drop ≥10, with HR rise ≥20 — reflects volume depletion (e.g.
  purging), not pure restriction (Ch. 2).
- The **"2-3-4 rule"** for normal electrolytes (Mg ≈ 2 mg/dL, phosphorus ≈
  3 mg/dL, potassium ≈ 4 mEq/L), hypokalemia risk below 3 mEq/L, and
  critical QTc prolongation ≥500 msec (Ch. 10).
- **Hepatitis of starvation**: AST/ALT elevation tracks degree of
  underweight/malnutrition, and levels ≥150 U/L independently predict
  hypoglycemia (Ch. 3).
- The book's broader critique that **BMI alone is a poor severity marker**
  is the whole reason the classifier needs weight-history and physiologic
  features, not just current BMI (Ch. 16).

**Bulimia nervosa (BN) and atypical BN** — `src/disorders/bulimia.py`
- Binge-and-purge at least once a week for 3+ months; patients are "not
  typically underweight, although they may be medically malnourished"
  (Ch. 7, p.4558-4559).
- **BN has no restricting subtype** — binge+purge is definitional. What
  varies is the **compensatory mechanism** (vomiting, laxative abuse,
  diuretic abuse, or excessive exercise, p.4550-4551), modeled as
  `compensatory_behavior_type`.
- **Exercise-type purging does not cause the same electrolyte physiology**
  as vomiting/laxatives/diuretics (p.4570-4571) — the generator's
  potassium/orthostatic logic scales down accordingly.
- **Atypical BN's differentiator from full BN is frequency, not weight
  status** (p.4563-4566) — a genuinely different split mechanism than
  atypical AN's weight-based one.

**Binge eating disorder (BED)** — `src/disorders/binge_eating.py`
- Recurrent binge episodes with loss of control and no compensatory
  behaviors, often at higher weight; Ch. 10 ("Binge Eating Disorder (BED)
  and Weight Stigma") argues these patients' medical complications are
  frequently misattributed to weight itself rather than worked up as an
  eating disorder.

**ARFID (Avoidant/Restrictive Food Intake Disorder)** — `src/disorders/arfid.py`
- Restrictive intake from sensory sensitivity or fear of aversive
  consequences (e.g. choking), **without** the body-image disturbance or
  fear of weight gain that define AN/AAN (Ch. 14). That absence is the
  actual DSM-5 exclusion criterion, and is built as an inversion of the
  AN/AAN feature pattern (high restriction, low weight/shape concern)
  rather than a lower-severity version of it.

**OSFED, other** — `src/disorders/osfed_other.py`
- A subthreshold catch-all per the book's general OSFED framing (Ch. 1,
  p.830-831), covering presentations that don't meet full criteria for any
  named disorder above.

**Supplementary population-level statistics** from peer-reviewed
literature: Sawyer et al., *Physical and psychological morbidity in
adolescents with atypical anorexia nervosa*, Pediatrics (2016); a
systematic review/meta-analysis of medical instability in typical vs.
atypical adolescent AN (PMC10080852), including the MEED risk-stratification
thresholds; Peebles et al., *Medical complications and management of
atypical anorexia nervosa*, J Eat Disord (2022); Hay et al. (2017) and
Lindvall Dahlgren et al. (2017) for population prevalence figures used in
the realistic-prevalence evaluation set; Hail & Le Grange (2018) for BN
subthreshold rates.

Note: the book itself doesn't publish a downloadable dataset — none exists
for these conditions outside of protected clinical records — so all
numeric distributions in the generator are original approximations
calibrated to match the ranges and relationships described in these
sources, not transcriptions of real patient data.

## Scope and exclusions

Deliberately **not** covered in this pass, despite being real conditions:

- **Pica** and **rumination disorder** as primary diagnoses — not covered
  in this book (verified: zero mentions of "pica" in the source text;
  rumination appears only as a GI-complication topic in Ch. 11, not as its
  own diagnostic chapter).
- **Diabulimia** (insulin restriction as purging in type 1 diabetics) — the
  book does cover this (Ch. 13), but it needs its own feature domain
  (insulin dosing, glucose logs, DKA risk) rather than reusing the
  electrolyte/vital feature set the other modules share.

All three are natural extension points: each disorder is a self-contained
module under `src/disorders/` implementing `generate_group(rng, age, sex,
height_cm, label) -> dict`, registered in `src/disorders/__init__.py`'s
`DISORDER_MODULES`. Adding one doesn't require touching any existing
module. See [Extending the project](#extending-the-project).

## Subtypes

AN's restricting-vs-binge/purge subtype and BN's compensatory-mechanism
variation are modeled as **input features** (`purging_behaviors`,
`binge_behaviors`, `compensatory_behavior_type`), not additional target
classes — the classifier sees them and can use them, but they don't expand
the label space. Note that BN has no DSM-5 "restricting subtype": binge and
purge are both required by definition.

## Repo structure

```
src/
  clinical_reference.py   shared reference_median_bmi(), used by both generation and features
  disorders/
    common.py              shared generation helpers (vitals/labs, exam findings, weight trajectory, etc.)
    anorexia.py             AN + atypical AN
    bulimia.py               BN + atypical BN
    binge_eating.py           BED
    arfid.py                    ARFID
    osfed_other.py                subthreshold catch-all
    healthy.py                     `none` baseline
    __init__.py                     DISORDER_MODULES registry, balanced/prevalence label distributions
  data_gen.py    orchestrator: samples demographics, dispatches to disorder modules -> data/synthetic_patients.csv
  features.py    derived clinical features + risk flags, sklearn preprocessing, LABELS (single source of truth)
  model.py       trains logistic regression + random forest, saves to models/
  evaluate.py    metrics, confusion matrices, feature importances -> reports/
```

`data/`, `models/`, and `reports/` are created by the pipeline at run time
and are gitignored — nothing in them is checked in, since it's all
regenerable from the code and a seed.

## Setup

Requires Python 3.10+ (uses `X | Y` union type hints and `dict | None`
defaults).

```bash
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
```

Dependencies: `numpy`, `pandas`, `scikit-learn`, `matplotlib`, `seaborn`,
`joblib` — see `requirements.txt` for version floors.

## Usage

```bash
python -m src.data_gen --n 6000 --out data/synthetic_patients.csv
python -m src.model --data data/synthetic_patients.csv
python -m src.evaluate --test-data data/test_set.csv
```

This generates 6,000 synthetic patients at the balanced training
distribution, trains both models on a stratified 75/25 split, and writes
evaluation plots to `reports/`.

Useful flags:
- `python -m src.data_gen --seed 7` — change the random seed for a
  different synthetic cohort (default 42, for reproducibility).
- `python -m src.model --test-size 0.3` — change the train/test split ratio.
- `python -m src.evaluate --report-dir reports/run2` — write plots
  somewhere other than `reports/`.

### Realistic-prevalence evaluation

The training distribution is deliberately balanced (roughly 45% `none`,
the rest spread across the seven ED classes) so every class has enough
examples to learn from — it does **not** reflect true population
prevalence (~1-4% combined for most of these conditions). To see how the
trained models perform under realistic class imbalance:

```bash
python -m src.data_gen --n 6000 --prevalence --out data/prevalence_patients.csv
python -m src.evaluate --test-data data/test_set.csv --prevalence-eval-data data/prevalence_patients.csv
```

This runs the *same trained* models against a separately generated,
prevalence-matched set and reports both sets of metrics side by side,
clearly labeled. Expect precision on the rare classes to drop substantially
in the prevalence-matched report — that's an honest picture of
deployment-time performance under real class imbalance, not a bug.

## Example results

From one run with `--n 6000` and the default seed (numbers will vary
slightly run to run and are illustrative, not a claim of fixed model
performance):

|                        | balanced accuracy | balanced AAN recall | prevalence-matched AAN recall |
|------------------------|:---:|:---:|:---:|
| logistic regression    | 94.7% | 0.85 | 0.81 |
| random forest          | 94.2% | 0.75 | 0.78 |

Both models hit macro-average ROC-AUC around 0.99 on the balanced set. The
confusion matrix's error mass concentrates almost entirely on the two
boundary pairs you'd expect from the clinical design rather than being
spread randomly: `anorexia_nervosa` ↔ `atypical_anorexia` (they share an
identical behavioral-severity distribution by construction and differ only
on weight status, which has deliberate real-world overlap near the
threshold) and `osfed_other` bleeding slightly into its neighbors (it's
built as a subthreshold catch-all). `binge_eating_disorder` and `arfid`
are essentially perfectly separated from everything else, because their
defining features (`loss_of_control_eating`/`binge_frequency_per_week` for
BED; `food_variety_count`/low weight-shape-concern for ARFID) don't
overlap with the restrictive/purging feature space the other classes live
in. Feature importance plots (`reports/feature_importance_*.png`) confirm
the top-ranked features are the clinically-motivated engineered ones —
`binge_frequency_per_week`, `weight_suppression_pct`, `pct_median_bmi`,
`restrictive_intake_score` — not something spurious.

## Reading the results

Because the classes are deliberately balanced in the training data,
accuracy alone isn't a meaningful summary — `evaluate.py` reports per-class
precision/recall/F1 and specifically calls out recall on the
`atypical_anorexia` class. Missing a true AAN case (a false negative) is
the exact failure mode the book is about — a patient who is "sick enough"
but gets dismissed because they're not underweight. A model that achieves
high accuracy by trading away AAN recall for AN/none precision would be
reproducing that clinical bias, not correcting it.

## Limitations

Worth being explicit about, since it's easy for a well-organized pipeline
to read as more authoritative than it is:

- **All data is synthetic.** No real patient was involved at any point.
  The generative distributions are calibrated to match ranges and
  relationships described in the cited literature, but they are still
  approximations authored for this project, not measurements.
- **`reference_median_bmi()` is a coarse linear approximation**, not the
  real CDC/WHO growth charts it stands in for (see
  `src/clinical_reference.py`). A production-grade version of this project
  would use the actual growth-chart tables.
- **The training class distribution is deliberately balanced**, not
  representative of true prevalence — that's why the prevalence-matched
  evaluation mode exists, but even that mode's prevalence figures
  (`src/disorders/__init__.py`'s `PREVALENCE_LABEL_PROBS`) are
  order-of-magnitude estimates from a handful of papers, not a rigorous
  epidemiological model.
- **No real-world validation.** There's no way to check these synthetic
  patients or this model's behavior against actual clinical outcomes,
  because doing so would require exactly the real patient data this
  project deliberately avoids using.
- **Feature engineering encodes one book's clinical framework.** Sick
  Enough is a well-regarded, physician-authored source, but it's one
  author's synthesis — thresholds like the QTc/BN-frequency/hypokalemia
  cutoffs used here reflect that source (supplemented by a handful of
  peer-reviewed papers), not a consensus guideline document.

None of this is disqualifying for a research/educational project — but it
is disqualifying for clinical use, which is why the disclaimer at the top
of this README is not boilerplate.

## Extending the project

To add a new disorder (e.g. diabulimia, pica — see
[Scope and exclusions](#scope-and-exclusions)):

1. Create `src/disorders/<name>.py` with a `generate_group(rng, age, sex,
   height_cm, label) -> dict` function, following the pattern in any
   existing module (e.g. `arfid.py` is a relatively short, self-contained
   example). Reuse helpers from `src/disorders/common.py`
   (`behavioral_score`, `weight_trajectory`, `vitals_and_labs`,
   `exam_findings`, etc.) wherever the new disorder's physiology overlaps
   with existing ones, and start from `common.baseline_row_defaults()` so
   every column in the shared schema gets a value.
2. If the new disorder needs raw columns no existing disorder uses (like
   `binge_frequency_per_week` was added for BN/BED), add them to
   `common.baseline_row_defaults()` with a population-normal/inapplicable
   default, and to `RAW_NUMERIC`/`RAW_BOOLEAN`/`CATEGORICAL` in
   `src/features.py`.
3. Add the new label to `LABELS` in `src/features.py` (the single source
   of truth other modules import from), then register the module in
   `DISORDER_MODULES` (and `LABEL_PROBS`/`PREVALENCE_LABEL_PROBS`) in
   `src/disorders/__init__.py`.
4. Add any new derived clinical flags to `add_clinical_features()` in
   `src/features.py`, cited to their clinical source the same way the
   existing flags are.
5. Regenerate data, retrain, and re-evaluate (see [Usage](#usage)) — no
   other file needs to change.

To add a new raw feature to an *existing* disorder (e.g. a new lab value),
add it to that disorder module's `generate_group()`, add a
population-normal default for it in `common.baseline_row_defaults()` so
other disorders still populate the column, and add it to the appropriate
list in `src/features.py`.
